# Artifact Filtering — Design Spec

## Проблема

Сейчас каждый агент получает **все артефакты всех зависимостей** целиком в промпте. Это приводит к огромным промптам (до 388 KB) и расходу на cache read (до 4.2M токенов на агента). При этом большая часть контекста не нужна — например, qa-engineer получает весь код frontend и backend, хотя ему достаточно API-контрактов.

## Решение

### 1. Формат данных — фильтры в edges

В `pipeline_graph.edges` добавляем опциональный третий элемент — список разрешённых артефактов:

```json
["system-architect", "backend-developer", ["api_contracts.md", "db_schema.md"]]
```

Правила:
- `["a", "b"]` — без фильтра = передавать всё (обратная совместимость)
- `["a", "b", ["file.md"]]` — передавать только указанные файлы
- `["a", "b", []]` — ничего не передавать в промпт

### 2. Оркестратор — agent_runner.py

`get_input_artifacts_for_agent()` — при наличии третьего элемента в edge, фильтрует артефакты зависимости по имени файла (endswith match).

`_build_full_prompt()` — для code-агентов (backend-developer, frontend-developer, devops-engineer, qa-engineer) добавляет `project_path` в промпт, чтобы они могли читать код из файловой системы.

### 3. Pipeline Builder — дефолтные фильтры

`pipeline_builder.py` задаёт разумные дефолты для стандартного графа. Документация передаётся в промпте, код — через ФС.

### 4. Dashboard — UI редактирование

На странице агента (`/project/[id]/agent/[name]`) добавляется секция "Входные артефакты" где видно:
- От какого агента-зависимости какие артефакты приходят
- Чекбоксы для включения/выключения каждого артефакта
- Сохранение через API в state

API endpoint: `POST /api/state/[id]/artifact-filters` — обновляет третий элемент в edges.

## Ожидаемый эффект

Для проекта to-do:
- frontend-developer: 388 KB → ~80 KB (только контракты + дизайн, без полного output PO/BA)
- qa-engineer: 206 KB → ~60 KB (контракты + структура, без кода)
- Экономия cache read: ~50-70% токенов
